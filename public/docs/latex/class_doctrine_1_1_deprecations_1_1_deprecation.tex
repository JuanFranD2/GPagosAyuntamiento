\doxysubsection{Detailed Description}
Manages Deprecation logging in different ways.

By default triggered exceptions are not logged.

To enable different deprecation logging mechanisms you can call the following methods\+:


\begin{DoxyItemize}
\item Minimal collection of deprecations via get\+Triggered\+Deprecations() \textbackslash{}\+Doctrine\textbackslash{}\+Deprecations\textbackslash{}\+Deprecation\+::enable\+Tracking\+Deprecations();
\item Uses @trigger\+\_\+error with E\+\_\+\+USER\+\_\+\+DEPRECATED \textbackslash{}\+Doctrine\textbackslash{}\+Deprecations\textbackslash{}\+Deprecation\+::enable\+With\+Trigger\+Error();
\item Sends deprecation messages via a PSR-\/3 logger \textbackslash{}\+Doctrine\textbackslash{}\+Deprecations\textbackslash{}\+Deprecation\+::enable\+With\+Psr\+Logger(\$logger);
\end{DoxyItemize}

Packages that trigger deprecations should use the {\ttfamily trigger()} or {\ttfamily trigger\+If\+Called\+From\+Outside()} methods.

Trigger a deprecation for the given package and identifier when called from outside.

"{}\+Outside"{} means we assume that \$package is currently installed as a dependency and the caller is not a file in that package. When \$package is installed as a root package then deprecations triggered from the tests folder are also considered "{}outside"{}.

This deprecation method assumes that you are using \doxylink{namespace_composer}{Composer} to install the dependency and are using the default /vendor/ folder and not a \doxylink{namespace_composer}{Composer} plugin to change the install location. The assumption is also that \$package is the exact composer packge name.

Compared to {\ttfamily \doxylink{}{trigger()}} this method causes some overhead when deprecation tracking is enabled even during deduplication, because it needs to call {\ttfamily \doxylink{}{debug\+\_\+backtrace()}}
